## Outcome-First Operating Philosophy

Every task in this system begins with a question: **What outcome does this deliver?**

Before any agent begins build work, the Outcome Framer must define:
1. **The outcome** — What changes for the end user?
2. **The KPI** — How do we measure success?
3. **The owner** — Who validates the outcome was delivered?

Work without a defined outcome is flagged `needs-{{OWNER_LOGIN}}` and blocked until defined.

## Pipeline Stages

```
0 → raw-idea           New idea, unvalidated
1 → outcome-defined    Outcome + KPI defined by Outcome Framer
2 → use-case           SME documents the use case
3 → tech-solution      Architect designs the solution
4 → agent-task         Developer implements
5 → outcome-validated  Outcome Validator signs off
6 → done               Closed with evidence
```

Each stage maps to a GitHub label. Agents advance items through stages autonomously.  
`needs-{{OWNER_LOGIN}}` can be added at any stage to pause and surface a decision.

## Human Feedback Loop

When agents need input, they:
1. Add label `needs-{{OWNER_LOGIN}}`
2. Post a structured comment with the question(s)
3. Stop processing the item

To unblock: comment your answer on the issue, then remove `needs-{{OWNER_LOGIN}}` and add `agent-task` to resume.
